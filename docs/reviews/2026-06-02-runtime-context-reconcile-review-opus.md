## §A — Independent re-verification of the three corrections (I did not take them on trust)

**(a) Canon-vs-CANON, not canon-vs-code — CONFIRMED.** `SDD_FIREGRID_UNIFIED_PRODUCTION_WIRING.md:12,14`: *"Phase 2 collapsed the substrate to three primitives (Workflow + DurableTable + Signal) … The substrate is settled."* `runtime-design-constraints.md:80-90`: keyed subscribers "run to completion, and return," **"No long-lived parked bodies,"** and — critically — the same doc is workflow-aware: *"No workflow-engine ban. `@effect/workflow` is valid execution machinery … the banned shape is a workflow body that represents the lifetime of an entity, parks across many events."* So the canon **deliberately** bans exactly the shipped body (`runtime-context.ts:113` parks on `Workflow.suspend` across the entity lifetime). Agreed it's a P0 doc reconcile — and (see §D2) amending canon is the *higher-bar* branch, since the constraint is deliberate and current, not stale.

**(b) `signal.ts` "Atomically" over-claims — CONFIRMED.** Comment `signal.ts:25-27`: *"Atomically: records the signal row, performs the optional companion row write, then arms."* Impl `writeSignalRow` (`signal.ts:168-180`): sequential `yield* insertSignalRow(...)` then `yield* options.write(...)` — two effects, no transaction. (Consistent with §A.c: the substrate has no atomic multi-row append — `DurableTable` is row-level.) The word "Atomically" should be "durably, in order" or the comment should say write+arm is *not* transactional.

**(c) F3 is only PARTIALLY a blocker — CONFIRMED; the proposal's §2.2 is wrong as written.** `DurableStreamsWorkflowEngine.test.ts:927` (`tf-e5rf`) proves a body point-reads its owned table, `Workflow.suspend`s when the row is absent, and a later table write + `engine.resume(executionId)` re-drives it to completion — **no DurableDeferred mailbox**. So "a row write cannot wake a subscriber / unproven" is false: *explicit-arm* table-wait is proven. What is absent is **automatic** table-write wakeup. The proposal must split these; as written §2.2 overstates the blocker and biases the decision toward "substrate can't do it."

**Leak-fix mechanism — Agent0 is right.** cancel/close already write a `kind:"terminal"` `SessionInputPayload` via `writeSessionInputSignal` (`channel-bindings.ts:296-300`, `:442` SessionCancel, `:465` SessionClose; span `firegrid.unified.session.terminal_signal`). The body consumes a terminal input to reach `reachedTerminal` and run its `adapter.deregister` activity, recording finalResult (`runtime-context.ts:131-153`). A **direct observer-side `adapter.deregister`** would `Scope.close` the registry (`codec-adapter.ts:518-529`) but leave the workflow execution **suspended with no finalResult** — an orphaned execution that recovery would try to re-arm. So the fix must route `Terminated` → terminal signal → body's existing `deregister`, **not** call deregister directly. Confirmed correct.

## §C/§D — Divergence and additions (the diversity value)

**§D1 — The A/B/C decomposition is a near-false trichotomy: a 4th option neither doc names.** A ("rewrite to substrate-push per-event subscribers") is gated on the two missing primitives (atomic multi-row append; automatic table-write wakeup). B ("amend canon to bless the entity-lifetime parked body"). C ("hybrid → resolves to B"). **Missing: per-turn run-to-completion.** The body **drains the currently-available input batch and returns** (sets finalResult) instead of parking for the entity lifetime; it is re-driven per input arrival by the *already-proven* explicit arm (`engine.resume`, `tf-e5rf`), with process continuity via the codec-adapter registry's no-op attach (`startOrAttach` returns early when the contextId is already registered — `codec-adapter.ts`, "the same process serves many inputs across attempts; the registry is the cross-attempt continuity"). This shape is **canon-COMPLIANT** ("runs to completion and returns") **without** either missing substrate primitive (explicit resume suffices) **and without** entity-lifetime parking (so **no canon amendment needed**). It is neither A (no new primitives required) nor B (no canon reversal). The PO's option set is currently biased toward "amend canon" by the omission of this middle. I de-risk one part of it from my own work: the cross-attempt drain boundary is **sequenced, not interleaved** (tf-0awo.26 / PR #822 — `deregister`'s `Scope.close` awaits the drain before the next attempt's first append), so per-turn/attempt-churn does not corrupt output ordering. **Calibration:** I have NOT proven this shape works end-to-end (see §D3); I present it as an option the trichotomy omits and that `tf-tvg1` should evaluate explicitly — not as the answer.

**§D2 — The B/C lean conflates MECHANISM with SHAPE (a pressure-test on a different axis than Agent0's §4c).** Agent0 argues signal.ts is too execution-specific to be a generic B arm. I argue a sharper point: the verified substrate gap — **no automatic table-write wakeup** — justifies an **explicit arm** (a *mechanism*: write+`resume`, C-shaped, already proven) but does **not** justify an **entity-lifetime parked body** (a *shape*). These are separable, and §D1 is the existence proof that you can keep the explicit arm *and* satisfy canon's no-park ban. So the inference "substrate can't push → therefore bless the parked body (B)" is a **non-sequitur**: the substrate fact compels a thin arm, not a parked shape. This is exactly where the coordinator's motivated-reasoning worry lands — the lean borrows a true substrate fact to justify a shape the fact does not require. Net: the C-mechanism (explicit arm) is source-justified; the B-shape (parked entity-lifetime body) is a *choice*, not substrate-compelled, and should be argued on its own merits (e.g., operational simplicity), not as "the substrate forces it."

**§D3 — A substrate fact neither doc checked, and it is load-bearing for §D1.** Both docs implicitly assume the entity-lifetime parked body is the *only* way to get multi-turn session continuity. The open question that decides whether §D1 is real or collapses: **does the agent's in-process session state survive across per-turn workflow executions?** The codec-adapter reuses one OS process across attempts (registry), but per-turn run-to-completion means a *new* workflow execution (attempt N+1) per turn — does the codec/agent session (e.g. ACP `sessionId`, conversation context) remain continuous when the body returns and a later execution re-attaches, or does anything in the per-execution path reset it? tf-0awo.26 proved drain *ordering* is safe across attempts; it did **not** prove session-state *continuity*. This is the single fact that should gate §D1, and neither doc raises it. It is also a precise, cheap addition to `tf-tvg1`'s matrix.

**§D4 — Order-of-operations: agree, with one fix.** "Fix P0 now (decoupled) → source-gate before any sim → one consolidated SDD off the verdict" is right, and Agent0's faster 3-question source gate is the correct economy (it already collapses pure-A). But the gate must test a **fourth arm** — "canon-compliant per-turn run-to-completion via the existing explicit arm" (§D1) — not just {push-auto / push+explicit / polling}. Without that arm the gate inherits the trichotomy's bias and steers the PO toward amending canon by omission.
