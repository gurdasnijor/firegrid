## Current Channel Shape

The SDD maps top-level prompt to `send(HostPrompt, payload)` and session prompt to `send(SessionPrompt, payload)` (`docs/sdds/SDD_FIREGRID_ONE_SUBSTRATE_PRIMITIVE.md:410`, `docs/sdds/SDD_FIREGRID_ONE_SUBSTRATE_PRIMITIVE.md:416`), and explicitly classifies both prompt channels as egress channels (`docs/sdds/SDD_FIREGRID_ONE_SUBSTRATE_PRIMITIVE.md:556`).

On main, the protocol channel contracts match that shape: `HostPromptChannelService` is `EgressChannel<typeof PublicPromptRequestSchema>`, while `SessionPromptChannelService.forSession(...)` returns `EgressChannel<typeof SessionHandlePromptInputSchema>` (`packages/protocol/src/channels/host-control.ts:57`, `packages/protocol/src/channels/host-control.ts:58`, `packages/protocol/src/channels/host-control.ts:64`, `packages/protocol/src/channels/host-control.ts:65`, `packages/protocol/src/channels/host-control.ts:68`). The generic `EgressChannel` binding only exposes `append(payload): Effect<void, ...>` (`packages/protocol/src/channels/core.ts:59`, `packages/protocol/src/channels/core.ts:65`, `packages/protocol/src/channels/core.ts:124`, `packages/protocol/src/channels/core.ts:128`, `packages/protocol/src/channels/core.ts:130`).

Both prompt Live Layers currently preserve that void egress shape by discarding the inserted row: `makeHostPromptChannel` calls `appendInputIntent(...).pipe(Effect.asVoid)`, and `makeSessionPromptChannelForSession` does the same for session-scoped prompts (`packages/protocol/src/launch/host-control-request.ts:125`, `packages/protocol/src/launch/host-control-request.ts:131`, `packages/protocol/src/launch/host-control-request.ts:132`, `packages/protocol/src/launch/host-control-request.ts:133`, `packages/protocol/src/launch/host-control-request.ts:137`, `packages/protocol/src/launch/host-control-request.ts:144`, `packages/protocol/src/launch/host-control-request.ts:152`). Host-sdk and client-sdk standalone layers both consume those same protocol factories (`packages/host-sdk/src/host/channels/host-control/index.ts:67`, `packages/host-sdk/src/host/channels/host-control/index.ts:139`, `packages/host-sdk/src/host/channels/host-control/index.ts:140`, `packages/client-sdk/src/channels/host-control-default.ts:24`, `packages/client-sdk/src/channels/host-control-default.ts:42`, `packages/client-sdk/src/channels/host-control-default.ts:43`).

## Option-A Implementation Plan

Recommended path: add an append-returning-row egress variant while preserving the public prompt return contracts.

1. Add a protocol-level return-bearing egress binding shape beside the existing void `EgressChannel`, not by changing the existing generic contract in place. The narrowest shape is an `AppendReturningTargetBinding<S, R>` plus `ReturningEgressChannel<S, R>` / factory, with prompt response schema `RuntimeInputIntentRowSchema`.
2. Change `HostPromptChannelService` to the return-bearing prompt egress service and `SessionPromptChannelService.forSession(...)` to return the session-scoped return-bearing egress service. Keep the target names (`host.prompt`, `session.prompt`) and request schemas unchanged.
3. Update `makeHostPromptChannel` and `makeSessionPromptChannelForSession` to return `appendInputIntent(...)` directly instead of `Effect.asVoid`, preserving context validation, `insertOrGet`, `createdAt`, `_otel`, and idempotent found-row behavior.
4. Rewire `firegrid.prompt` and `session.prompt` to call the channel binding and return the row. Rewire `firegrid.sessions.prompt` through the host prompt channel as well, then preserve its current tool output by mapping the returned row to `{ appended: true, sessionId, inputId: row.intentId }`.
5. Delete the local `appendRuntimeInputIntent` helper once all three prompt call sites use channel bindings; keep `decode*` helpers and error mapping local to the client projection.
6. Tests to run after implementation: `pnpm --filter @firegrid/client-sdk test`, `pnpm --filter @firegrid/client-sdk typecheck`, and full `pnpm preflight`.

If Gurdas picks option (b), the breaking surface is `FiregridService.prompt` and `FiregridSessionHandle.prompt`, whose current signatures return `RuntimeInputIntentRow`. If option (c) wins, the request/response schemas can reuse the option-a response schema, but prompt becomes `CallableChannel` instead of an egress variant; that would diverge from the current SDD rows cited above.
