# Durability Assumption Audit

Date: 2026-05-16

Scope: neutral inventory of current durable row-family assumptions for the workflow-native runtime substrate exploration. This audit does not select a substrate direction and does not recommend which state should become ephemeral.

Note: the brief listed `packages/protocol/src/runtime-ingress/table.ts`, but this checkout has the runtime-ingress table declaration in `packages/protocol/src/runtime-ingress/schema.ts`.

Read classifications used below:

- Live-only: needed only by currently running fibers/subscribers.
- Cross-restart soft: improves replay, observability, or idempotency, but the immediate runtime can still make progress with degraded history.
- Cross-restart hard: a restart/reconstruction path explicitly reads the row family to resume, route, dedupe, or complete previously committed work.

## 4. Minimum-durability surface

Cross-restart hard rows observed in current code:

- RuntimeContext: required for host ownership, stream routing, start/prompt/read validation, and external attach (`packages/runtime/src/host/internal/runtime-context-helpers.ts:24`, `packages/client/src/firegrid.ts:509`).
- RuntimeRunEvent: required to allocate attempts without resetting in-progress runs (`packages/runtime/src/authorities/runtime-control-plane-recorder.ts:71`).
- RuntimeEvent: required by tool router replay from committed ToolUse rows and by public output wait/snapshot surfaces (`packages/runtime/src/agent-event-pipeline/subscribers/tool-router.ts:29`, `packages/client/src/firegrid.ts:634`).
- RuntimeIngressInput: required for pending input delivery, tool-result duplicate guards, and client-visible input history (`packages/runtime/src/agent-event-pipeline/subscribers/ingress-delivery.ts:33`, `packages/runtime/src/agent-event-pipeline/subscribers/tool-router.ts:75`).
- RuntimeIngressDelivery: required for at-most-once stdin/input delivery after claim (`packages/runtime/src/agent-event-pipeline/authorities/runtime-ingress-delivery-tracker.ts:34`, `packages/runtime/test/sources/sandbox/local-process-stdin-delivery.test.ts:159`).
- Wait and WaitCompletion: required for wait reattachment and crash reconciliation (`packages/runtime/src/durable-tools/internal/wait-router.ts:216`, `packages/runtime/src/durable-tools/internal/reconcile.ts:29`).
- WorkflowExecution, WorkflowActivity, WorkflowDeferred, WorkflowClockWakeup: required for execution replay/final results, activity replay, deferred completion, and clock recovery (`packages/runtime/src/workflow-engine/internal/engine-runtime.ts:92`, `packages/runtime/src/workflow-engine/internal/engine-runtime.ts:196`, `packages/runtime/src/workflow-engine/internal/engine-runtime.ts:244`, `packages/runtime/src/workflow-engine/internal/engine-runtime.ts:83`).
- WorkflowActivityClaim: required for durable concurrent-worker exclusion; impact is topology-dependent but current tests assert durability for raced workers (`packages/runtime/src/workflow-engine/internal/engine-runtime.ts:41`, `packages/runtime/test/workflow-engine/DurableStreamsWorkflowEngine.test.ts:489`).

Cross-restart soft rows observed in current code:

- RuntimeLogLine: currently external snapshot/debug history; no audited runtime resume path depends on log rows (`packages/client/src/firegrid.ts:547`, `packages/client/src/firegrid.ts:478`).

Live-only row families: none observed. Every current row family has at least one cross-restart or external read.

## 9. Specific durability-reduction opportunities

These are opportunities for later exploration, not recommendations:

- RuntimeLogLine appears to be external observability/history rather than runtime resume state. Exploration could ask whether log durability needs the same retention and replay guarantees as output events, while preserving the snapshot contract or documenting any change (`packages/client/src/firegrid.ts:478`, `packages/client/src/firegrid.ts:547`).
- RuntimeRunEvent has two distinct uses: public status/history and internal attempt allocation. Exploration could separate the minimal attempt-allocation state from richer run history if a future design wants different retention classes (`packages/runtime/src/authorities/runtime-control-plane-recorder.ts:71`, `packages/client/src/firegrid.ts:481`).
- RuntimeEvent has both external observation and internal tool-router replay duties. Exploration could distinguish committed tool-dispatch inputs from general user-visible event history, provided deterministic tool-result replay remains intact (`packages/runtime/src/agent-event-pipeline/subscribers/tool-router.ts:29`, `packages/runtime/test/subscribers/tool-router.test.ts:118`).
- RuntimeIngressInput has pending-delivery, duplicate-guard, and external history responsibilities. Exploration could separate pending operational inputs from historical inputs only if prompt/session/permission APIs and snapshots keep their expected behavior (`packages/runtime/src/agent-event-pipeline/subscribers/ingress-delivery.ts:33`, `packages/client/src/firegrid.ts:861`).
- WorkflowActivityClaim is topology-dependent in this audit. Exploration could identify whether deployments that guarantee single-worker execution need the same durable claim surface as multi-worker execution, while retaining the current raced-worker behavior where applicable (`packages/runtime/src/workflow-engine/internal/engine-runtime.ts:219`, `packages/runtime/test/workflow-engine/DurableStreamsWorkflowEngine.test.ts:514`).

## 10. Open questions and uncertainties

- The brief referenced `packages/protocol/src/runtime-ingress/table.ts`, but this checkout has runtime-ingress table definitions in `packages/protocol/src/runtime-ingress/schema.ts`; this audit treats that schema file as the source of truth.
- This audit did not read the forbidden substrate proposal documents, so it intentionally does not map findings onto any named substrate path.
- This audit classifies row families by current code and tests. It does not prove that every external consumer in downstream applications depends on every exposed field.
- RuntimeLogLine has no audited runtime-resume dependency, but it is externally visible through snapshots; changing retention would be an API/observability question rather than a pure runtime question.
- WorkflowActivityClaim is conditionally essential because the hard requirement appears when concurrent workers can race. Current tests cover that topology, but single-worker-only deployments may have a different minimum surface.
